🔄 严格消息流程设计

确保每个步骤完成后才进行下一步的聊天体验

📋 新的严格流程

为了确保数据一致性和用户体验的可预测性，我们实现了一个严格的步骤化流程，每个步骤必须完成后才能进行下一步。

🔢 详细步骤流程

1
用户发送消息
用户输入消息并点击发送，消息立即渲染在聊天界面中
↓
2
上传用户消息到数据库
显示"💾 正在保存您的消息..."，等待数据库存储完成
↓
3
AI开始响应
用户消息保存成功后，AI开始生成回复，显示"正在思考中..."
↓
4
实时渲染AI回复
AI流式回复过程中，实时更新AI消息气泡内容
↓
5
上传AI消息到数据库
AI回复完成后，显示"💾 正在保存AI回复..."，等待数据库存储完成
↓
6
用户可以继续聊天
AI消息保存成功后，移除存储提示，用户可以发送下一条消息

⏱️ 交互时间线

T0: 用户操作
用户输入"你好"并点击发送 立即完成

T6: 存储AI消息
AI回复完成，显示"💾 正在保存AI回复..."，调用 /api/messages/create 等待完成
T7: 完成
AI消息保存成功，移除存储提示，用户可以继续聊天 可以继续

💻 关键实现代码

1. 用户消息存储

// 步骤2: 上传用户消息到数据库 setIsStoringMessages(true); onUpdate({ content: "💾 正在保存您的消息...", role: "assistant" }); const userStorageSuccess = await storeMessage( currentSessionId, "user", message.content, { stage: "user_input", timestamp: Date.now() } ); if (!userStorageSuccess) { setIsStoringMessages(false); onError(new Error("用户消息保存失败，请重试")); return; }

2. AI响应处理

// 步骤3: 用户消息保存成功，开始AI响应 onUpdate({ content: "正在思考中...", role: "assistant" }); // 调用AI流式API const response = await fetch("/api/chat/stream", { method: "POST", headers: { "Content-Type": "application/json", Authorization: `Bearer ${token}`, }, body: JSON.stringify({ session_id: currentSessionId, query: message.content, }), });

if (data.finished) { // 步骤6: AI回复完成，上传AI消息到数据库 onUpdate({ content: fullContent + "\n\n💾 正在保存AI回复...", role: "assistant" }); const aiStorageSuccess = await storeMessage( currentSessionId, "assistant", fullContent, finalWorkflowState || { stage: "ai_response", timestamp: Date.now() } ); if (aiStorageSuccess) { // 步骤7: AI消息保存成功，移除存储提示 onUpdate({ content: fullContent, role: "assistant" }); console.log("对话完成，用户可以继续聊天"); } // 重置存储状态，允许用户继续聊天 setIsStoringMessages(false); }
❌ 旧流程（批量存储）
  • 用户发送消息 → 立即显示
  • AI实时回复
  • 回复完成后批量存储用户+AI消息
  • 存储期间用户无法继续聊天
  • 存储失败影响整个对话
✅ 新流程（严格步骤）
  • 用户发送消息 → 立即显示
  • 先存储用户消息 → 成功后AI才响应
  • AI实时回复
  • AI完成后立即存储AI消息
  • 存储完成后用户才能继续

🎯 严格流程的优势

数据一致性：确保每条消息都被正确存储
错误处理：任何步骤失败都会阻止后续操作
用户反馈：每个步骤都有明确的状态提示
防止冲突：用户必须等待当前对话完成才能继续
可预测性：用户清楚知道系统正在做什么
调试友好：每个步骤都可以独立监控和调试
⚠️ 注意事项
  • 网络延迟：每个存储步骤都需要网络请求，可能增加总体响应时间
  • 用户等待：用户需要等待存储完成才能继续，可能影响流畅性
  • 错误恢复：任何步骤失败都会中断整个流程，需要用户重试
  • 状态管理：需要精确管理 isStoringMessages 状态，避免状态不一致
